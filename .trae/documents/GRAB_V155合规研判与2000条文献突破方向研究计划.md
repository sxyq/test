## 一、项目概述

### 1.1 赛题

百度 2026 CTI 生成式推荐广告排序推理性能优化赛道。模型架构：

| 模块 | 规格 |
|------|------|
| RepEncoder | Embedding 5M×512×28 slots |
| TransformerEncoder | 8 层, 8 头, d_model=512, dim_ff=1024 |
| SMoE | 8 Experts, Top-2 Gating |
| CTRModel | 输出层 |

### 1.2 评估指标

- AUC ∈ [0.65, 1]，PCOC ∈ [0.85, 1.15]
- 推理时间 < 300s
- `score = score_latency × 70 + score_model × 30`

### 1.3 当前状态

| 指标 | Baseline | 当前合规最佳 V139 | 违规最高 V121 |
|------|----------|-------------------|---------------|
| latency | 229.18s | 45.90s | 41.58s |
| score_all | 25.85 | 68.33 | 69.34（违规） |
| AUC | 0.759 | 0.752 | 0.752 |
| PCOC | 1.110 | 1.059 | 0.990 |

距满分 100 差 31.67 分，剩余差距几乎全在 `score_latency`。

### 1.4 核心规则（剪枝相关）

> **不可修改**模型组网和相关参数（违规 0 分）
> **允许**量化、稀疏、剪枝优化
> **剪枝规则**：不能简单粗暴地"直接剪掉几层"，可以删掉对输出贡献较小、冗余度较高、结构上可安全移除的部分。即**要求结构不能被破坏**

## 二、V155 合规研判结论

### 2.1 V153 FFN 神经元剪枝：✅ 合规

**实现位置**：[infer.py](file:///Users/sunyiyang/Desktop/Project/Baidu%20%20GRAB/versions/55_v155/V155-DUAL-PRUNE-sxyq/infer.py) 第 676-711 行（`SMoE._v109_prepare_dense_weights()` 内）

**机制**：SMoE Expert 隐层维度剪枝，dim_ff 1024→768（ratio=0.25），使用 `||fc1[row]|| * ||fc2[:, col]||` 重要性度量，每 Expert 独立 top-k 选择。

**合规理由**：
1. dim_ff 是 FFN 内部隐层宽度，非模型组网架构参数（d_model/n_heads/num_layers 才是）
2. 8 个 Expert、Top-2 门控、fc1→ReLU→fc2 计算流、残差连接、LayerNorm 全部完整保留
3. 学术界公认的标准结构化神经元剪枝（SliceGPT/LLM-Pruner/FLAP 同类）
4. 不改变计算逻辑，PCOC 可通过 bias 烘焙零开销修正

### 2.2 V155 head_dim 剪枝：⚠️ 高风险/潜在违规

**实现位置**：[infer.py](file:///Users/sunyiyang/Desktop/Project/Baidu%20%20GRAB/versions/55_v155/V155-DUAL-PRUNE-sxyq/infer.py) 第 1053-1089 行（`load_model()` 内）

**机制**：注意力头维度剪枝，head_dim 64→48（ratio=0.25），使用 `||qkv_w|| * ||out_w||` 重要性度量，直接修改 `seq_enc.head_dim = new_head_dim`。

**违规理由**：
1. **修改模型组网参数**：`head_dim = d_model // n_heads = 64` 是由核心组网参数派生的架构参数，V155 直接篡改为 48，违反"不可修改模型组网和相关参数"
2. **破坏注意力核心计算**：缩放因子 √64→√48、QK^T 维度 64→48、每头表征 64→48、输出总维度 8×48=384≠512
3. **学术共识不支持**：LASP 论文（ICLR 2026）明确"不建议剪枝 Q/K/V 的子维度"，会导致与位置编码机制不一致；V155 缺乏 CORP 论文要求的闭式补偿
4. **历史判例警示**：V116-V131 跳层被判违规的核心是"破坏结构"，head_dim 剪枝同样改变注意力结构（粒度更细但性质类似），V120/V121 即使 PCOC 修正回区间仍被判违规

### 2.3 处置建议

| 剪枝项 | 判定 | 处置 |
|--------|------|------|
| V153 FFN 剪枝 | ✅ 合规 | 保留使用，配合 bias 烘焙 |
| V155 head_dim 剪枝 | ⚠️ 潜在违规 | **不提交**，通过 `GRAB_V155_PRUNE_HEAD_DIM=0` 关闭 |
| 替代：整头剪枝 | ✅ 合规 | 若需剪枝注意力，移除整个 head（8→6），保持 head_dim=64 |
| 替代：SVD 低秩分解 | ✅ 合规 | 数学严谨，不改变 head_dim |

### 3.4 V100 (SM70) 兼容性关键修正

**重要发现**：V100 (compute capability 7.0) 的 INT8 支持与 Turing (7.5) 不同：

| 技术 | V100 (SM70, CC 7.0) | Turing (SM75, CC 7.5) | Ampere+ (SM80+) |
|------|---------------------|----------------------|------------------|
| FP16 Tensor Core | ✅ 125 TFLOPS | ✅ | ✅ |
| INT8 dp4a 点积 | ✅ 有限支持 | ✅ | ✅ |
| INT8 Tensor Core (mma) | ❌ 无 | ✅ | ✅ |
| BF16 | ❌ | ❌ | ✅ |
| FP8 | ❌ | ❌ | ❌(SM90+) |
| FlashAttention-2 官方 | ⚠️ 声称支持可能回退 | ✅ | ✅ |
| xformers | ✅ 原生 | ✅ | ✅ |
| Triton | ✅ | ✅ | ✅ |
| CUDA Graph | ✅ | ✅ | ✅ |

**结论**：V100 上 INT8 W8A8 只能用 dp4a 软件路径，加速 1.3-1.5x（非理论 2x）。FP16 + xformers + Triton grouped GEMM 是 V100 最可靠的加速组合。

## 六、假设与决策

### 6.1 关键假设

1. **V100 是评测硬件**：README 提及 AI Studio GPU，V100 (SM70) 兼容性是核心约束
2. **本机 Mac 无法运行推理**：所有验证必须在 AI Studio 云端进行
3. **flash-attn 可能未生效**：V139 的 45.90s 可能是 SDPA 回退而非真 flash-attn
4. **INT8 dp4a 加速有限**：V100 无 INT8 Tensor Core，量化收益打折
5. **Triton 可用**：PyTorch 2.x 自带 Triton，V100 支持

### 6.2 关键决策

| 决策 | 选择 | 理由 |
|------|------|------|
| V155 是否提交 | **不提交** | head_dim 剪枝潜在违规 |
| 注意力后端 | **xformers 优先** | SM70 原生支持最可靠 |
| MoE 内核 | **Triton grouped GEMM 优先** | V100 可用，消除 4x 过计算 |
| 量化方案 | **SmoothQuant W8A8（dp4a）** | V100 有 dp4a，加速打折但仍有效 |
| 剪枝方案 | **V153 FFN 剪枝保留** | 合规，配合 bias 烘焙 |
