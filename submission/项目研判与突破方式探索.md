## 五、当前性能瓶颈分析

### 5.1 距离满分（100分）的差距

- 当前合规最佳 score_all = **68.33**（V139），距满分 100 差 **31.67 分**。
- 得分公式：`score_all = score_latency × 70 + score_model × 30`。
  - V139 的 score_latency ≈ 0.84（45.90s），score_model ≈ 0.30（AUC/PCOC 达标但未提升）。
  - score_model 上限约 0.30（AUC 不可提升，仅守门），即模型分贡献约 9 分，已接近上限。
  - **剩余差距几乎全部在 score_latency**：要接近满分需 latency 显著下降（score_latency→1.0 需 latency 极低）。
- 违规最高分 V121 = 69.34（跳2层），仅比 V139 高 1.01 分，说明**合规优化空间有限，需新方向突破**。

### 5.4 当前推理时间的主要瓶颈（模块耗时）

综合 README 模型架构与 `.trae` 文档分析，45.90s（V139）的主要耗时模块：

1. **SMoE Dense 计算（最大瓶颈）**：V109/V139 采用 Dense batched SMoE，8 个 Expert 全算但只用 Top-2，存在 **4x 过计算**。这是合规优化最大空间，MegaBlocks/ScatterMoE 块稀疏内核可消除（预期 -5~15s）。
2. **注意力计算**：当前 SDPA / flash-attn 路径。V100 上 flash-attn 可能静默回退 SDPA，xformers memory_efficient_attention 是 SM70 最可靠替代（预期 -2~5s）。
3. **RepEncoder embedding 查表**：5M vocab × 28 slots，已用 FP16 + sparse embedding 优化，token-major 布局可减少 copy（V98/V102 本地验证正向，待 A800 实测）。
4. **Kernel launch 开销**：2039 个 batch 逐个处理，CUDA Graph 可消除（V149 待测，预期 -1~3s）。
5. **PCOC 校准开销**：bias 烘焙是零开销修正（V120 验证），但跳层等改变计算逻辑的修正会增加 5s+（V119/V121）。

## 六、核心结论

1. **当前合规最佳**：V139（flash-attn），45.90s / score 68.33 / AUC 0.752 / PCOC 1.059，距满分差 31.67 分。
2. **违规最高分**：V121（跳2层+精确修正）69.34 分，仅高 1.01 分，且已被判违规，说明合规优化需新方向突破而非继续微调。
3. **V153/V154/V155 剪枝版本无线上提交记录**，均为"待测"状态。V153 FFN 剪枝合规可继续，V155 head_dim 剪枝高风险建议放弃。
4. **最大瓶颈**：SMoE Dense 计算的 4x 过计算（合规优化最大空间），其次是 V100 上 flash-attn 可能未真生效。
5. **已用尽方向**：PyTorch 微优化、torch.compile、block-diagonal mask、Gate 优化、层剪枝、用户 batch 拆分等均已验证无效或违规。

7. **关键硬件发现**：V100 (SM70) 不支持 FlashAttention-2 官方内核与 FP8，但原生支持 INT8 DP4A、xformers、Triton、CUDA Graph——这些是合规且 V100 可用的加速路径。

### 2.2 "结构不能被破坏"的具体定义

规则对"结构不能被破坏"的定义通过**反面举例**给出：
- **违规示例**："直接剪掉几层"（即跳层/层剪枝）——这是"简单粗暴"的，破坏结构
- **合规示例**："删掉对输出贡献较小、冗余度较高、**结构上可安全移除**的部分"

关键区分点在于：
1. **跳层**（移除整个 Transformer 层）= 破坏结构 = 违规
2. **移除冗余部分**（神经元/通道/头等细粒度结构，且贡献小、可安全移除）= 合规

规则同时强调"**不可修改模型组网和相关参数**"，这是判定 head_dim 剪枝合规性的核心条款。

## 三、V153 FFN 剪枝的具体实现方式（保留结构完整性的方式）

### 3.1 实现位置与机制

**位置**：`V155-DUAL-PRUNE-sxyq/infer.py` 第 676-711 行（在 `SMoE._v109_prepare_dense_weights()` 方法内）。V153 和 V154 中实现完全相同，仅 `V153_PRUNE_RATIO` 默认值不同（V153/V155 为 0.25，V154 为 0.50）。

**核心代码逻辑**（第 680-711 行）：

```python
if V153_PRUNE_FFN and not self.training:
    E = self.num_experts
    dim_ff = self._all_fc1_w.shape[2]  # current dim_ff (1024)
    keep_count = int(dim_ff * (1.0 - V153_PRUNE_RATIO))  # 768 for 25% pruning
    if keep_count < dim_ff:
        with torch.no_grad():
            # 重要性度量：每个隐层神经元的连接权重范数乘积
            fc1_norms = self._all_fc1_w.norm(dim=1)  # [E, dim_ff]
            fc2_norms = self._all_fc2_w.norm(dim=2)  # [E, dim_ff]
            importance = fc1_norms * fc2_norms  # [E, dim_ff]

            # 每个 expert 独立选择 top-k 神经元
            keep_indices = importance.topk(keep_count, dim=1).indices.sort(dim=1).values

            # 剪枝 fc1: [E, D, dim_ff] → [E, D, keep_count]
            self._all_fc1_w = self._all_fc1_w.gather(2, idx_expanded).contiguous()
            self._all_fc1_b = self._all_fc1_b.gather(1, keep_indices).contiguous()

            # 剪枝 fc2: [E, dim_ff, D] → [E, keep_count, D]
            self._all_fc2_w = self._all_fc2_w.gather(1, idx_expanded2).contiguous()
```

### 3.2 结构完整性保留方式

V153 FFN 剪枝通过以下方式保留结构完整性：

| 结构要素 | 剪枝前 | 剪枝后 | 是否保留 |
|----------|--------|--------|----------|
| Expert 数量 | 8 | 8 | ✅ 完整 |
| Top-2 门控 | 是 | 是 | ✅ 完整（gate 权重未动） |
| Expert 内部计算流 | fc1→ReLU→fc2 | fc1→ReLU→fc2 | ✅ 完整 |
| d_model（输入/输出维度） | 512 | 512 | ✅ 完整 |
| dim_ff（隐层宽度） | 1024 | 768 | ⚠️ 缩窄但结构不变 |
| 残差连接 | 是 | 是 | ✅ 完整 |
| LayerNorm | 是 | 是 | ✅ 完整 |

**关键点**：dim_ff 是 FFN/Expert 的**内部隐层宽度**，并非模型组网的"架构参数"（如 d_model、n_heads、num_layers）。缩小 dim_ff 等价于移除冗余神经元，Expert 的"输入→隐层→输出"三段式结构完整保留。这与学术界标准的**结构化神经元剪枝**（Structured Neuron Pruning）完全一致。

### 3.3 重要性度量原理

V153 采用 `||fc1[row]|| * ||fc2[:, col]||` 作为神经元重要性度量——即一个隐层神经元的重要性由其**输入连接权重范数**与**输出连接权重范数的乘积**决定。这是经典的"连接权重乘积"剪枝准则，移除重要性低的神经元对输出贡献最小，符合规则中"删掉对输出贡献较小、冗余度较高"的要求。

## 四、V155 head_dim 剪枝的具体实现方式

### 4.1 实现位置与机制

**位置**：`V155-DUAL-PRUNE-sxyq/infer.py` 第 1053-1089 行（在 `load_model()` 函数内，`model.to(dev)` 之后、`_v109_prepare_dense_weights()` 之前）。

**核心代码逻辑**：

```python
if V155_PRUNE_HEAD_DIM:
    seq_enc = model.seq_encoder
    n_heads = seq_enc.n_heads           # 8
    old_head_dim = seq_enc.head_dim     # 64
    d_model = seq_enc.d_model           # 512
    new_head_dim = int(old_head_dim * (1.0 - V155_HEAD_DIM_RATIO))  # 48
    if new_head_dim < old_head_dim and new_head_dim > 0:
        with torch.no_grad():
            for i in range(seq_enc.num_layers):
                qkv = seq_enc.qkv_proj[i]
                outp = seq_enc.out_proj[i]
                # QKV weight: [3*d_model, d_model] → [3, n_heads, head_dim, d_model]
                qkv_w = qkv.weight.data.view(3, n_heads, old_head_dim, d_model)
                # OutProj weight: [d_model, d_model] → [d_model, n_heads, head_dim]
                out_w = outp.weight.data.view(d_model, n_heads, old_head_dim)
                # 重要性度量：每个 head_dim 索引的连接权重范数乘积
                qkv_imp = qkv_w.norm(dim=(0, 1, 3))  # [old_head_dim]
                out_imp = out_w.norm(dim=(0, 1))     # [old_head_dim]
                importance = qkv_imp * out_imp
                keep = importance.topk(new_head_dim).indices.sort().values  # [48]
                # 剪枝 QKV weight 和 bias
                qkv_w_new = qkv_w[:, :, keep, :].contiguous().view(-1, d_model)
                qkv.weight.data = qkv_w_new
                qkv.bias.data = qkv.bias.data.view(3, n_heads, old_head_dim)[:, :, keep].contiguous().view(-1)
                # 剪枝 OutProj weight（bias 不变，是 [d_model]）
                out_w_new = out_w[:, :, keep].contiguous().view(d_model, -1)
                outp.weight.data = out_w_new
            # 更新 head_dim 参数
            seq_enc.head_dim = new_head_dim  # 64 → 48
```

### 4.2 对模型结构的实际影响

V155 的 head_dim 剪枝对模型结构产生了**深层改变**，远超"移除冗余部分"范畴：

| 结构要素 | 剪枝前 | 剪枝后 | 改变性质 |
|----------|--------|--------|----------|
| head_dim | 64 | 48 | **核心架构参数改变** |
| n_heads | 8 | 8 | 不变 |
| d_model | 512 | 512 | 不变 |
| qkv_proj 权重形状 | [1536, 512] | [1152, 512] | 输出维度改变 |
| out_proj 权重形状 | [512, 512] | [512, 384] | 输入维度改变 |
| QK^T 矩阵每头维度 | [S, 64]×[64, S] | [S, 48]×[48, S] | 注意力计算维度改变 |
| 注意力缩放因子 | √64 = 8 | √48 ≈ 6.93 | **缩放因子改变**（见第 532 行 `d = q.size(-1)`） |
| 每头表征维度 | 64 | 48 | 表征能力改变 |
| 注意力输出总维度 | 8×64=512 | 8×48=384 | **与 d_model 不再匹配**，需 out_proj 补偿 |

### 4.3 关键问题：head_dim 是模型组网参数

在 `TransformerEncoder.__init__`（第 793-800 行）中：

```python
def __init__(self, d_model, n_heads, num_layers, dim_ff, ...):
    self.d_model = d_model           # 512
    self.n_heads = n_heads           # 8
    self.head_dim = d_model // n_heads  # 64 ← 由 d_model 和 n_heads 派生
    assert d_model % n_heads == 0
```

`head_dim = d_model // n_heads = 512 // 8 = 64` 是由两个核心组网参数（d_model=512, n_heads=8）**派生的架构参数**。V155 在第 1084 行直接修改 `seq_enc.head_dim = new_head_dim`（48），这本质上是在运行时**篡改了模型组网参数**。

### 4.4 forward 中的连锁影响

剪枝后，`TransformerEncoder.forward`（第 816-853 行）的行为发生改变：

```python
qkv = self.qkv_proj[i](x)                              # 输出 [B,S,1152] 而非 [B,S,1536]
qkv = qkv.view(B, S, self.n_heads, 3 * self.head_dim)  # [B,S,8,144] 而非 [B,S,8,192]
q, k, v = torch.split(qkv, self.head_dim, dim=-1)      # 每个 [B,S,8,48] 而非 [B,S,8,64]
# 注意力计算中 d = q.size(-1) = 48，缩放因子从 √64 变为 √48
attn_out = attn_out.permute(0, 2, 1, 3).reshape(B, S, self.n_heads * self.head_dim)  # [B,S,384] 而非 [B,S,512]
x = residual + self.out_proj[i](attn_out)              # out_proj: [384→512] 而非 [512→512]
```

### 6.1 判例详情（来自提交记录）

### 6.2 违规原因分析

README.md 第 126-128 行明确记录：

> **V116-V131全部使用层剪枝（跳层），违反比赛规则"结构不能被破坏"。**
> 虽然跳层+bias烘焙路线最高达到37.52s/score=69.81，但该方向已被判定违规，后续版本必须遵守合规方向。
> 核心教训：任何改变计算逻辑的优化（跳层、Gate优化等）都可能破坏PCOC校准，导致零分。

违规的核心原因：
1. **跳层 = 移除整个 Transformer 层** = "直接剪掉几层" = 规则明确禁止的"简单粗暴"行为
2. **破坏了模型的层级结构**：8 层 → 7 层（或更少），模型深度这一核心组网参数被改变
3. **PCOC 校准被破坏**：跳层导致 logit 分布偏移，PCOC 偏出 [0.85, 1.15] 区间
4. **即使 bias 烘焙能修正 PCOC，结构破坏本身仍是违规**：V120/V121 虽然通过 bias 烘焙将 PCOC 修正回区间内并取得高分，但仍被判违规

### 6.3 判例的合规性边界启示

V116-V131 判例确立了以下合规边界：
- **移除整个层（改变 num_layers）= 违规**（即使 PCOC 可修正）
- **改变计算逻辑（如 Gate 优化 V136）= 违规**（PCOC 崩到 0.09）
- **移除冗余神经元/通道（不改变层级结构）= 合规**（规则明确允许）

## 七、学术界对 head_dim 剪枝（Q/K/V 子维度剪枝）的共识

### 7.1 本地资源情况

本地 `literature/` 目录的 23 篇 PDF 中**没有任何专门讨论结构化剪枝的论文**。搜索结果确认无 LASP、CORP、SPRINT、SliceGPT、LLM-Pruner、FLAP 等剪枝论文的本地 PDF。`.trae/documents/` 中的研判文档引用的 LASP 和 CORP 均为**在线资源**（ICLR 2026 投稿/录用），非本地文件。

### 7.2 学术共识（基于 .trae 研判文档引用的在线资源）

根据 `.trae/documents/GRAB_V155合规研判与2000条文献突破方向研究.md` 第 79-83 行的引用：

1. **LASP 论文（ICLR 2026 投稿）**明确指出：
   > "we do not recommend pruning sub-dimensions of Q/K/V"（不建议剪枝 Q/K/V 的子维度）
   
   原因：剪枝 Q/K/V 的子维度会导致**与位置编码机制不一致**。head_dim 的每个维度可能编码特定的位置/语义信息，移除部分维度会破坏这种编码一致性。

2. **标准的"注意力头剪枝"（Attention Head Pruning）**指的是**移除整个 head**（如 8→6 个头），而非缩小每个 head 的维度。这是学术界公认的结构化剪枝方式（如 Voita et al. 2019, Michel et al. 2019）。

3. **CORP 论文（ICLR 2026）**虽然剪枝 attention 维度，但需要**闭式补偿**（ridge regression 恢复表征能力）。V155 **没有这种补偿机制**，直接移除维度后依赖 out_proj 的线性映射"吸收"维度变化，这在理论上不严谨。

### 7.3 head_dim 剪枝与标准结构化剪枝的区别

| 剪枝类型 | 操作对象 | 学术界共识 | 对结构的影响 |
|----------|----------|------------|--------------|
| **层剪枝** | 整个 Transformer 层 | 违规/破坏结构 | 改变 num_layers |
| **头剪枝** | 整个 attention head | 公认合规的结构化剪枝 | 改变 n_heads，但每个剩余 head 的 head_dim 不变 |
| **FFN 神经元剪枝** | FFN 隐层神经元 | 公认合规的结构化剪枝 | 缩小 dim_ff，结构不变 |
| **head_dim 剪枝** | Q/K/V 的子维度 | **LASP 明确不推荐** | 改变 head_dim，影响注意力缩放、QK^T 维度、位置编码一致性 |

## 八、合规性研判结论

### 8.1 V153 FFN 剪枝：合规 ✅

**判定：合规**

理由：
1. **符合规则字面**：dim_ff（1024→768）是 FFN 的内部隐层宽度，并非模型组网的"架构参数"（d_model、n_heads、num_layers 才是）。缩小 dim_ff 属于"删掉对输出贡献较小、冗余度较高、结构上可安全移除的部分"。
2. **结构完整保留**：8 个 Expert、Top-2 门控、fc1→ReLU→fc2 计算流、残差连接、LayerNorm 全部完整保留，仅隐层变窄。
3. **学术界公认**：FFN 神经元剪枝是标准的结构化剪枝方法，广泛见于 SliceGPT、LLM-Pruner、FLAP 等论文。
4. **不改变计算逻辑**：门控权重未动，Expert 选择逻辑不变，仅每个 Expert 的隐层计算量减少。
5. **PCOC 风险可控**：可通过 bias 烘焙（V153_LOGIT_BIAS）零开销修正 PCOC 偏移。

### 8.2 V155 head_dim 剪枝：高风险/潜在违规 ⚠️

**判定：高风险，潜在违规，不建议提交**

理由：

**（1）规则字面违规风险**

`head_dim = d_model // n_heads = 512 // 8 = 64` 是由两个核心组网参数（d_model=512, n_heads=8）**派生的架构参数**。V155 在第 1084 行直接修改 `seq_enc.head_dim = 48`，这属于"**修改模型组网和相关参数**"——规则第 14 行明确规定"不可修改模型组网和相关参数（违规 0 分）"。

**（2）超出"安全移除冗余部分"范畴**

head_dim 剪枝不仅移除了冗余维度，还改变了注意力的核心计算特性：
- **注意力缩放因子改变**：√64=8 → √48≈6.93（第 532 行 `d = q.size(-1)`），这直接改变了 softmax 分布的锐度
- **QK^T 矩阵维度改变**：每个头的内积从 64 维降至 48 维
- **每头表征能力改变**：从 64 维降至 48 维
- **注意力输出总维度与 d_model 不匹配**：8×48=384 ≠ 512，需 out_proj 从 [512,512] 改为 [512,384] 来补偿

这些改变远超"安全移除冗余部分"的范畴，属于**重构注意力机制的核心架构**。

**（3）学术共识不支持**

LASP 论文（ICLR 2026 投稿）明确"不建议剪枝 Q/K/V 的子维度"，因为会导致与位置编码机制不一致。V155 缺乏 CORP 论文所要求的闭式补偿机制。

**（4）历史判例的警示**

V116-V131 跳层被判违规的核心原因是"破坏结构"。head_dim 剪枝同样改变了注意力的结构（虽然粒度更细），风险类似。特别是 V120/V121 即使通过 bias 烘焙将 PCOC 修正回区间内并取得高分，仍被判违规——这说明**结构破坏本身是违规的，与 PCOC 是否可修正无关**。

**（5）PCOC 风险**

head_dim 改变会直接影响注意力分布（缩放因子变化），进而影响 logit 校准，PCOC 可能偏移出 [0.85, 1.15] 区间。虽然 V155 保留了 V153_LOGIT_BIAS 烘焙机制，但 head_dim 剪枝对 PCOC 的影响比 FFN 剪枝更难预测和修正。

### 8.3 综合建议

| 剪枝项 | 合规判定 | 建议 |
|--------|----------|------|
| **V153 FFN 神经元剪枝** | ✅ 合规 | 可继续使用，配合 bias 烘焙修正 PCOC |
| **V155 head_dim 剪枝** | ⚠️ 高风险/潜在违规 | **不建议提交**，高风险被判违规导致 0 分 |
| **替代方案：整头剪枝** | ✅ 合规（学术界公认） | 若要剪枝注意力，应改为移除整个 head（8→6 个头），保持 head_dim=64 不变 |
| **替代方案：SVD 低秩分解** | ✅ 合规 | 用 SVD 分解 QKV/OutProj 权重，数学上更严谨，不改变 head_dim |

### 8.4 最终结论

**V153 合规，V155 的 head_dim 剪枝部分潜在违规。** V155 作为"DUAL-PRUNE"版本，其 FFN 剪枝部分（继承自 V153）合规，但新增的 head_dim 剪枝部分存在高风险。如果提交 V155，建议通过环境变量 `GRAB_V155_PRUNE_HEAD_DIM=0` 关闭 head_dim 剪枝，仅保留 FFN 剪枝（此时等同于 V153）。

## 六、本地知识中关于 V100 (SM70) 兼容性的关键发现

### 6.1 V100 硬件特性

- **SM70 架构**，16GB HBM2
- **FP16 Tensor Core**：支持，FP16 GEMM 理论峰值 125 TFLOPS（FP32 的 8x）
- **无 BF16 支持**：必须使用 FP16
- **INT8 Tensor Core**：仅支持 `dp4a` 指令（点积），不如 Turing/Ampere 的 `mma` 指令高效
- **无 INT4 Tensor Core**：需软件反量化
- **无 FP8 支持**：FlashAttention-3 的 FP8 特性不可用
- **80 个 SM**：batch=1, heads=8 时仅 8 个并行任务远未填满

### 6.2 各优化在 V100 上的兼容性

| 优化 | V100 兼容性 | 关键限制 |
|---|---|---|
| FP16 推理 | ✅ 完全兼容 | d_model=512 满足 8 倍数对齐要求；低频 Embedding ID 可能丢精度，建议 AMP autocast 而非 `.half()` 预转换 |
| FlashAttention-2 | ✅ 兼容（最低支持 SM70） | 必须使用 FP16（无 BF16）；FP16 注意力可能溢出，但 FA2 内部 FP32 累积，安全 |
| FlashAttention-3 | ❌ 不兼容 | 需要 WGMMA、TMA、FP8，均为 H100 特性；可借鉴 cp.async 预取思路（1.1-1.2x） |
| SmoothQuant W8A8 | ⚠️ 有限兼容 | INT8 Tensor Core 仅 `dp4a`，实际加速 1.5-2x（非理论 2-4x） |
| INT-FlashAttention | ⚠️ 有限兼容 | INT8 操作需 `dp4a` 指令，性能不如 Ampere+；建议优先 FA2 FP16，INT8 作为第二阶段 |
| DQRM INT4 Embedding | ⚠️ 软件实现 | V100 无 INT4 Tensor Core，需软件反量化，计算开销可能抵消带宽收益 |
| MiLo 3-bit 量化 | ❌ 不兼容 | V100 不支持 INT8 Tensor Core GEMM（需 Turing+），3-bit 更无硬件支持；低秩补偿思路可用于 INT8 精度恢复 |
| torch.compile | ⚠️ 需验证 | 本地 Inductor 受限（V88/V89），需 A800 能力门控；动态 shape 问题需 `dynamic=True` 或 padding |
| CUDA Graph | ✅ 兼容 | capture 期间不能有 CPU-GPU 同步、动态内存分配、条件分支；SMoE 动态路由需 Branch Compaction |
| channels_last | ✅ 兼容 | V100 上 FP16 + channels_last 比 contiguous 快 1.2-1.4x；但 Embedding 查表不支持，仅对 Transformer/SMoE 生效 |

### 6.3 V100 关键注意事项汇总

1. **FP16 是最安全高效的优化**：V100 Tensor Core 原生支持，d_model=512 满足对齐
2. **INT8 收益打折**：`dp4a` 指令效率不如 `mma`，实际加速 1.5-2x 而非 2-4x
3. **INT4/FP8 不可用**：需软件实现或放弃
4. **FlashAttention-2 是 V100 上最佳注意力方案**：FA3 不可用，FA1 已被 FA2 取代
5. **Tensor Core 对齐要求**：FP16 对齐 8（d_model=512 已满足），INT8 对齐 16（需注意）
6. **输出层保持 FP32**：最终 logits 建议保持 FP32 确保排序精度
7. **Embedding 量化策略**：建议分步 FP16 → INT8 → INT4，逐 slot 测试敏感度
8. **本机无法运行推理**（Mac 无 NVIDIA GPU），所有 V100 测试必须在 AI Studio 云端进行

### 一、TopKGate.forward 返回值对比

**三个版本完全一致**，代码相同：

```python
# 三个版本相同的 TopKGate.forward
def forward(self, x):
    logits = self.w_g(x)                          # [B,S,E]
    if self.noisy_gating and self.training:
        logits = logits + torch.randn_like(logits) * 0.1
    probs = torch.softmax(logits, dim=-1)          # [B,S,E] — 全 softmax
    topk_score, topk_idx = torch.topk(probs, self.k, dim=-1)  # [B,S,k]
    return topk_idx, topk_score, probs
```

| 属性 | V129 (V139代理) | V154 | V157A |
|------|-----------------|------|-------|
| `topk_score` 含义 | **原始 softmax 概率**（未重新归一化） | 同左 | 同左 |
| `topk_score` dtype | float (跟随模型精度，FP16/FP32) | 同左 | 同左 |
| `topk_idx` dtype | **int64**（torch.topk 默认） | 同左 | 同左 |
| 返回顺序 | `(topk_idx, topk_score, probs)` | 同左 | 同左 |
| 是否重新归一化 | **否** — topk_score 之和 < 1 | 同左 | 同左 |

**关键发现**：`topk_score` 是对全部 8 个 expert 做 softmax 后取 top-2 的原始概率值，**没有**对 top-2 重新归一化。这意味着 gate 权重之和严格小于 1（等于 top-2 softmax 概率之和）。三个版本在此点完全一致，不存在差异。

def _v109_prepare_dense_weights(self):
    if self._v109_dense_weights_ready:
        return
    self._all_fc1_w = torch.stack([e.fc1.weight.t() for e in self.experts])  # [E, D, dim_ff]
    self._all_fc1_b = torch.stack([e.fc1.bias for e in self.experts])        # [E, dim_ff]

### 三、SMoE.forward 中 V109 Dense 路径对比

**三个版本的 V109 Dense 路径核心逻辑完全一致**：

```python
# 三个版本相同的 V109 Dense 路径
x_flat = x.reshape(-1, D)                    # [NS, D]
NS = B * S
gate_weights = x_flat.new_zeros(NS, self.num_experts)           # [NS, E] 全零
gate_weights.scatter_(1, topk_idx.reshape(NS, self.k), topk_score.reshape(NS, self.k))  # 散布 top-k 概率

# 批量 GEMM：所有 token 通过所有 expert
hidden = torch.matmul(x_flat.unsqueeze(0), fc1_w) + self._all_fc1_b.unsqueeze(1)  # [E, NS, dim_ff]
hidden = F.relu(hidden)
out_all = torch.matmul(hidden, fc2_w) + self._all_fc2_b.unsqueeze(1)               # [E, NS, D]

# 加权求和：gate_weights 转置后广播
out = (out_all * gate_weights.t().unsqueeze(-1)).sum(dim=0)  # [NS, D]
out = out.reshape(B, S, D)
```

**scatter 方式分析**：
- `gate_weights` 初始为 `[NS, E]` 全零矩阵
- `scatter_(dim=1, index=topk_idx[NS,k], src=topk_score[NS,k])` 将 top-k 概率值写入对应 expert 列
- 非 top-k expert 的权重保持为 0
- 最终 `gate_weights.t().unsqueeze(-1)` 变为 `[E, NS, 1]`，与 `out_all[E, NS, D]` 逐元素相乘后 sum(dim=0)

**V154/V157A 的额外逻辑**：在 V109 Dense 路径中增加了 V147 INT8 反量化分支（默认不激活）：
```python
if self._v147_int8_active:
    fc1_w = self._v147_fc1_int8.to(x_flat.dtype) * self._v147_fc1_scale.to(x_flat.dtype)
    fc2_w = self._v147_fc2_int8.to(x_flat.dtype) * self._v147_fc2_scale.to(x_flat.dtype)
else:
    fc1_w = self._all_fc1_w
    fc2_w = self._all_fc2_w
```

### 四、V157 稀疏路径完整逻辑（仅 V157A 和 V157-TRITON-MOE 有）

文件 `/Users/sunyiyang/Desktop/Project/Baidu  GRAB/submission/V157A-SPARSE-ONLY-sxyq/infer.py` 第 763-810 行：

```python
if V157_SPARSE_SMOE and not self.training and self._v109_dense_weights_ready:
    x_flat = x.reshape(-1, D)                # [NS, D]
    NS = B * S

    # 展平 topk: [NS, k] -> [NS*k]
    topk_idx_flat = topk_idx.reshape(-1)     # [NS*k]
    topk_score_flat = topk_score.reshape(-1) # [NS*k]
    token_ids = torch.arange(NS, device=x.device).repeat_interleave(self.k)  # [NS*k] = [0,0,1,1,2,2,...]

    # 按 expert id 排序分组
    order = torch.argsort(topk_idx_flat)
    expert_idx_sorted = topk_idx_flat[order]
    token_idx_sorted = token_ids[order]
    score_sorted = topk_score_flat[order]

    # 每个 expert 的 token 计数（一次 CPU-GPU 同步，循环内无同步）
    counts = torch.bincount(expert_idx_sorted, minlength=self.num_experts)
    starts = torch.cumsum(counts, dim=0) - counts
    counts_list = counts.tolist()    # 预取到 Python list
    starts_list = starts.tolist()

    # 获取权重（INT8 反量化或直接使用）
    if self._v147_int8_active:
        fc1_w = self._v147_fc1_int8.to(x_flat.dtype) * self._v147_fc1_scale.to(x_flat.dtype)
        fc2_w = self._v147_fc2_int8.to(x_flat.dtype) * self._v147_fc2_scale.to(x_flat.dtype)
    else:
        fc1_w = self._all_fc1_w
        fc2_w = self._all_fc2_w

    out_flat = x_flat.new_zeros(NS, D)
    for i in range(self.num_experts):
        count = counts_list[i]
        if count == 0:
            continue
        start = starts_list[i]
        cur_token_idx = token_idx_sorted[start:start + count]
        cur_score = score_sorted[start:start + count].unsqueeze(-1).to(x_flat.dtype)  # [count, 1]
        selected_x = x_flat.index_select(0, cur_token_idx)                             # [count, D]
        hidden = torch.matmul(selected_x, fc1_w[i]) + self._all_fc1_b[i]              # [count, dim_ff]
        hidden = F.relu(hidden)
        expert_out = torch.matmul(hidden, fc2_w[i]) + self._all_fc2_b[i]              # [count, D]
        out_flat.index_add_(0, cur_token_idx, expert_out * cur_score)

    out = out_flat.reshape(B, S, D)
```

**`index_add_` 使用方式分析**：
- `out_flat` 初始为 `[NS, D]` 全零
- 对每个 expert i，将该 expert 处理的 token 输出 `expert_out * cur_score` 累加到 `out_flat` 的对应 token 行
- 同一个 token 被路由到 2 个 expert，因此会在 2 次 expert 迭代中被 `index_add_` 累加——这是正确的 MoE 聚合行为
- `cur_score` 显式转换为 `x_flat.dtype`，避免 FP16/FP32 混算

**与原始稀疏路径（else 分支）的对比**：

| 特性 | V157 稀疏路径 | 原始稀疏路径 (else) |
|------|--------------|-------------------|
| token_ids 构造 | `arange(NS).repeat_interleave(k)` | `arange(NS).unsqueeze(1).expand(-1,k).reshape(-1)` |
| 结果等价性 | **完全相同**（都产生 `[0,0,1,1,...]`） | — |
| CPU-GPU 同步 | **1 次**（`counts.tolist()` + `starts.tolist()` 预取） | **8 次**（每 expert 调 `.item()` 2 次） |
| 权重来源 | 预堆叠的 `_all_fc1_w[i]`（可能已剪枝） | 原始 `self.experts[i]`（未剪枝） |
| score dtype 处理 | 显式 `.to(x_flat.dtype)` | 不转换（可能 FP32*FP16 自动提升） |
| 数学等价性 | **等价**（相同 gate 概率 × 相同 expert 输出） | — |

### 五、V153 FFN 剪枝实现对比

V153 剪枝在 V154 和 V157A 中代码完全相同（V129 无此功能），差异仅在默认剪枝比例：

**剪枝逻辑**（V154 第 676-711 行 / V157A 第 720-755 行）：
```python
# 重要性指标：fc1 和 fc2 连接权重的范数乘积
fc1_norms = self._all_fc1_w.norm(dim=1)   # [E, dim_ff]
fc2_norms = self._all_fc2_w.norm(dim=2)   # [E, dim_ff]
importance = fc1_norms * fc2_norms         # [E, dim_ff]

# 每 expert 选 top-k 神经元
keep_indices = importance.topk(keep_count, dim=1).indices.sort(dim=1).values  # [E, keep_count]

# 剪枝 fc1: [E, D, dim_ff] → [E, D, keep_count]
self._all_fc1_w = self._all_fc1_w.gather(2, idx_expanded).contiguous()
self._all_fc1_b = self._all_fc1_b.gather(1, keep_indices).contiguous()

# 剪枝 fc2: [E, dim_ff, D] → [E, keep_count, D]
self._all_fc2_w = self._all_fc2_w.gather(1, idx_expanded2).contiguous()
# fc2 bias [E, D] 不受 dim_ff 剪枝影响
```

**剪枝后权重 shape 一致性验证**（无 shape 不匹配问题）：

| 权重 | 剪枝前 | 剪枝后 (V154, 50%) | 剪枝后 (V157A, 25%) |
|------|--------|-------------------|-------------------|
| `_all_fc1_w` | [8, 512, 1024] | [8, 512, 512] | [8, 512, 768] |
| `_all_fc1_b` | [8, 1024] | [8, 512] | [8, 768] |
| `_all_fc2_w` | [8, 1024, 512] | [8, 512, 512] | [8, 768, 512] |
| `_all_fc2_b` | [8, 512] | [8, 512]（不变） | [8, 512]（不变） |

在 V157 稀疏路径中：
- `matmul(selected_x[count,D], fc1_w[i][D,keep])` → `[count, keep]` ✓
- `matmul(hidden[count,keep], fc2_w[i][keep,D])` → `[count, D]` ✓

在 V109 Dense 路径中：
- `matmul(x_flat[1,NS,D], fc1_w[E,D,keep])` → `[E, NS, keep]` ✓
- `matmul(hidden[E,NS,keep], fc2_w[E,keep,D])` → `[E, NS, D]` ✓

**结论：V153 剪枝不会导致权重 shape 不匹配问题**。所有路径的矩阵乘法维度在剪枝后仍然一致。

### 六、发现的差异和潜在 Bug

#### 1. V154 失败的可能原因：剪枝比例过于激进

V154 使用 `V153_PRUNE_RATIO = 0.50`（第 46 行），将 dim_ff 从 1024 砍到 512（50% 剪枝）。这远比 V157A 的 25% 激进，极可能导致 AUC/PCOC 严重退化。提交记录中 V116/V117 的层剪枝导致 PCOC 崩塌的前例说明，该模型对结构修改非常敏感。

#### 2. V157 稀疏路径的隐式依赖：V109_DENSE_SMOE 必须为 1

V157 稀疏路径的条件是：
```python
if V157_SPARSE_SMOE and not self.training and self._v109_dense_weights_ready:
```

而 `_v109_dense_weights_ready` 只在 `_v109_prepare_dense_weights()` 中被设为 True。该方法只在模型加载时被调用：
```python
if V109_DENSE_SMOE:   # ← 如果设为 0，则不会调用！
    for moe in model.seq_encoder.moe:
        moe._v109_prepare_dense_weights()
```

**潜在 Bug**：如果用户设置 `GRAB_V109_DENSE_SMOE=0` 但 `GRAB_V157_SPARSE_SMOE=1`（意图是"只用稀疏路径，不用 dense"），则 `_v109_dense_weights_ready` 保持 False，V157 稀疏路径不会执行，会回退到原始稀疏路径（使用未剪枝的 `self.experts[i]`）。这在 V157A 的默认配置中不会触发（两者都默认为 "1"），但属于设计陷阱。

#### 3. V147 INT8 + V153 剪枝的崩溃 Bug（潜伏，默认不触发）

在 `_v109_prepare_dense_weights()` 中，执行顺序为：
1. 堆叠权重 → `_all_fc1_w`、`_all_fc2_w`
2. V147 INT8 量化 → **删除** `_all_fc1_w`、`_all_fc2_w`（第 710-711 行）
3. V153 剪枝 → 访问 `self._all_fc1_w.shape[2]`（第 727 行）→ **AttributeError: _all_fc1_w**

由于 V147_INT8_SMOE 默认为 `"0"`，此 Bug 不会在默认配置中触发。但如果同时开启 V147 和 V153，代码会崩溃。

#### 4. V157 稀疏路径 vs V109 Dense 路径的数值等价性

V157 稀疏路径在数学上与 V109 Dense 路径**完全等价**：
- V109 Dense：计算全部 8 个 expert 输出，非 top-k 的 gate 权重为 0，乘以 0 后求和
- V157 Sparse：只计算 top-2 expert 输出，直接累加

唯一的微小差异来自浮点精度：V109 Dense 中 `0 * expert_out` 可能产生极小的非零值（denormal），但这对最终结果影响可忽略。

#### 5. V157 稀疏路径的 dtype 处理更安全

V157 稀疏路径显式将 `cur_score` 转换为 `x_flat.dtype`：
```python
cur_score = score_sorted[start:start + count].unsqueeze(-1).to(x_flat.dtype)
```

而原始稀疏路径不转换：
```python
cur_weight = expert_score_sorted[start:end]  # 可能是 FP32（softmax 输出）
```

如果模型在 FP16 模式下运行，`topk_score`（来自 softmax）可能是 FP32，原始路径中 `expert_out(FP16) * cur_weight(FP32)` 会自动提升为 FP32，然后 `index_add_` 到 FP16 的 `out_flat` 会隐式截断。V157 路径的显式转换避免了这种隐式行为，是更安全的做法。

#### 6. V157-HEAD-PRUNE 版本缺少 V157 稀疏路径

文件 `/Users/sunyiyang/Desktop/Project/Baidu  GRAB/versions/57_v157/V157-HEAD-PRUNE-sxyq/infer.py` 虽然在 V157 目录下，但**没有** `V157_SPARSE_SMOE` 环境变量和稀疏路径，只有 V109 Dense 和原始稀疏路径。这说明 V157 系列内部存在分叉：TRITON-MOE 和 SPARSE-ONLY 有稀疏路径，HEAD-PRUNE 没有。

### 八、核心结论

1. **三个版本的 TopKGate 完全一致**：`topk_score` 是原始 softmax 概率（未重新归一化），`topk_idx` 是 int64。不存在 gate 输出差异。

2. **V154 失败的主因是 V153 剪枝比例 50% 过于激进**（dim_ff 1024→512），而非 SMoE forward 逻辑本身有 bug。SMoE 代码在剪枝后 shape 仍然一致，不存在 shape 不匹配问题。

#### 2.1 每层 Attention 部分的 FLOPs

每层 Attention 的计算步骤（第742-753行）：

| 操作 | 形状 | FLOPs |
|------|------|-------|
| LayerNorm1 | [1, NS, 512] | 2 * NS * 512 (mean+var) |
| QKV 投影 | [1, NS, 512] @ [512, 1536] | 2 * NS * 512 * 1536 = **1,572,864 * NS** |

| Softmax + Attn*V | [1, 8, NS, NS] @ [1, 8, NS, 64] | 2 * 8 * NS * NS * 64 = **1,024 * NS^2** |
| Out 投影 | [1, NS, 512] @ [512, 512] | 2 * NS * 512 * 512 = **524,288 * NS** |

**Attention 总计**: ~2,097,152 * NS + 2,048 * NS^2

对于典型的 NS~5000（50用户 * ~100条/用户），NS^2 项约 51M FLOPs，线性项约 10.5M FLOPs。**Attention 在小 NS 下不是瓶颈**，因为 NS^2 项相对较小。

| 操作 | 形状 | FLOPs |
|------|------|-------|
| Gate 投影 | [NS, 512] @ [512, 8] | 2 * NS * 512 * 8 = **8,192 * NS** |
| Batched fc1 | [1, NS, 512] @ [8, 512, 1024] | 2 * 8 * NS * 512 * 1024 = **8,388,608 * NS** |
| ReLU | [8, NS, 1024] | ~8 * NS * 1024 |
| Batched fc2 | [8, NS, 1024] @ [8, 1024, 512] | 2 * 8 * NS * 1024 * 512 = **8,388,608 * NS** |
| Gate 加权求和 | [8, NS, 512] * [8, NS, 1] + sum | ~8 * NS * 512 |

**Dense SMoE 总计**: ~16,785,408 * NS（**4x 过计算**，因为 top-k=2 只需 2/8=25% 的计算量）

**原始稀疏路径**（第664-696行）：

| 操作 | 形状 | FLOPs |
|------|------|-------|
| Gate 投影 | 同上 | 8,192 * NS |
| 8 次 expert fc1 | 各 ~NS/4 * 512 * 1024 | 2 * 2 * NS * 512 * 1024 = **2,097,152 * NS** |
| 8 次 expert fc2 | 各 ~NS/4 * 1024 * 512 | 2 * 2 * NS * 1024 * 512 = **2,097,152 * NS** |

**稀疏 SMoE 理论 FLOPs**: ~4,202,496 * NS（仅为 Dense 的 1/4）

#### 2.3 瓶颈判断

| 组件 | 每层 FLOPs (NS=5000) | 占比 |
|------|---------------------|------|
| Attention (V139 Flash) | ~31M | 5% |
| SMoE Dense | ~84M | 95% |
| SMoE Sparse (理论) | ~21M | 68% |

**结论: SMoE 是绝对瓶颈**，占每层计算量的 68%-95%。Dense 路径虽然 FLOPs 是稀疏的 4 倍，但实际运行更快（见下节分析）。

### 3. V109 Dense 路径 vs 原始稀疏路径性能差异

#### 3.1 V109 Dense 路径（第647-663行）

```python
# 1. 构建 gate_weights: [NS, 8] 稀疏矩阵
gate_weights = x_flat.new_zeros(NS, self.num_experts)
gate_weights.scatter_(1, topk_idx.reshape(NS, self.k), topk_score.reshape(NS, self.k))

# 2. 2 次 batched GEMM（8 个 expert 同时计算）
hidden = torch.matmul(x_flat.unsqueeze(0), self._all_fc1_w) + self._all_fc1_b.unsqueeze(1)
hidden = F.relu(hidden)
out_all = torch.matmul(hidden, self._all_fc2_w) + self._all_fc2_b.unsqueeze(1)

# 3. gate 加权求和
out = (out_all * gate_weights.t().unsqueeze(-1)).sum(dim=0)
```

- **2 次 batched GEMM**: `[1, NS, 512] @ [8, 512, 1024]` 和 `[8, NS, 1024] @ [8, 1024, 512]`
- 每次是 8 个 `[NS, 512] @ [512, 1024]` 的 GEMM，**cuBLAS batched GEMM 高效调度**
- **零 Python 循环**，零 CPU-GPU 同步
- **代价**: 4x 过计算（8 个 expert 全算，但 gate_weights 让非 top-k 的贡献为 0）

#### 3.2 原始稀疏路径（第664-696行）

```python
for i in range(self.num_experts):  # 8 次 Python 循环
    count = int(counts[i].item())    # CPU-GPU 同步 #1
    if count == 0:
        continue
    start = int(starts[i].item())    # CPU-GPU 同步 #2
    end = start + count
    cur_token_idx = token_idx_sorted[start:end]
    cur_weight = expert_score_sorted[start:end]
    selected_x = x_flat.index_select(0, cur_token_idx)  # 小 GEMM 准备
    expert_out = self.experts[i](selected_x)             # 小 GEMM
    out_flat.index_add_(0, cur_token_idx, expert_out * cur_weight)
```

- **8 次 Python for 循环**，每次循环内：
  - 2 次 `.item()` 调用 → **CPU-GPU 同步**（CUDA stream 阻塞等待 GPU 完成）
  - 1 次 `index_select` → GPU 上的 gather 操作
  - 1 次小 GEMM `[count, 512] @ [512, 1024]` → **小矩阵乘法效率极低**
  - 1 次 `index_add_` → scatter 操作

#### 3.3 V100 上的性能对比

**实际测试结果**: V109 Dense = 48.31s vs 稀疏路径（Baseline）= 229.18s。Dense 快 **4.7 倍**。

**Dense 更快的原因**:

1. **GEMM 效率**: V100 的 Tensor Core 对 FP16 GEMM 的峰值算力为 125 TFLOPS，但需要矩阵足够大才能接近峰值。Dense 路径的 batched GEMM `[NS, 512] @ [512, 1024]`（NS~5000 时为 5000x512 @ 512x1024）可以充分利用 Tensor Core。稀疏路径的每个 expert 只有 ~NS/4=1250 行，小 GEMM 的 Tensor Core 利用率极低。

2. **CPU-GPU 同步**: 稀疏路径每层 8 expert x 2 次 `.item()` = 16 次同步/层，8 层 = **128 次同步/forward pass**。每次同步 ~0.5-1ms（取决于 GPU 队列深度），总计 ~64-128ms/forward pass。

3. **Python 循环开销**: 8 次 Python for 循环的 kernel launch 延迟，每次 ~10-50us，8 层 x 8 expert = 64 次循环，额外 ~1-3ms/forward pass。

4. **内存访问模式**: `index_select` + `index_add_` 是非连续内存访问，对 GPU 缓存不友好。Dense 路径的全矩阵乘法是连续访问。

**结论**: 在 V100 上，Dense 路径的 4x 过计算远小于小 GEMM + CPU-GPU 同步 + Python 循环的开销，因此 Dense 明显更快。

#### 5.1 同步点定位

#### 5.2 同步次数计算

- **稀疏 SMoE 路径**: 每层 8 expert x 2 次 `.item()` = 16 次同步
- **8 层**: 16 x 8 = **128 次同步 / forward pass**
- **~2039 个 batch**: 128 x 2039 = **~261,000 次同步 / 整个推理**

#### 5.3 同步开销估算

每次 `.item()` 的开销取决于 GPU 队列状态：
- **GPU 队列空时**: ~5-10us（几乎无等待）
- **GPU 队列满时**: ~0.5-2ms（需等待前面的 kernel 完成）
- **典型情况**: 在推理循环中，GPU 队列通常有积压，每次 `.item()` ~0.3-1ms

保守估计每次同步 0.5ms：
- 每 forward pass: 128 x 0.5ms = **64ms**
- 2039 个 batch: 64ms x 2039 = **~130s**

乐观估计每次同步 0.1ms：
- 每 forward pass: 128 x 0.1ms = **12.8ms**
- 2039 个 batch: 12.8ms x 2039 = **~26s**

#### 5.4 综合开销分析

Baseline（原始稀疏路径）总推理时间 229.18s，分解如下：

| 开销来源 | 估算时间 | 说明 |
|----------|----------|------|
| Attention (8层) | ~30-40s | SDPA + mask 构建 |
| SMoE GEMM (8层) | ~60-80s | 小 GEMM 效率低 |
| `.item()` 同步 (128次/pass) | ~26-130s | 取决于 GPU 队列深度 |
| Python 循环 + index_select/add | ~10-20s | kernel launch + 非连续访问 |
| 其他 (LayerNorm, residual, etc.) | ~5-10s | |

**关键发现**: `.item()` 同步开销可能在 26-130s 之间，这是 Baseline 229s 中不可忽视的组成部分。V109 Dense 路径通过消除所有 `.item()` 调用和 Python 循环，将这部分开销归零，同时用 batched GEMM 替代小 GEMM，这就是从 229s 降到 48s 的主要原因。

**核心原理**：

对 Expert 的 fc1/fc2 权重做 SVD 分解，将一次大 GEMM 拆成两次小 GEMM：

```
原始:  fc1 = x @ W1          # W1: [D, dim_ff],    FLOPs = NS × D × dim_ff
SVD:   fc1 = x @ U_R @ V_R   # U_R: [D, R], V_R: [R, dim_ff]
                             # FLOPs = NS × R × (D + dim_ff)
```

**计算量对比**（假设 D=256, dim_ff=1024, R=128）：

| | 原始 | SVD (R=128) | 减少 |
|--|------|-------------|------|
| fc1 FLOPs | NS×256×1024 | NS×128×(256+1024) | **50%** |
| fc2 FLOPs | NS×1024×256 | NS×128×(1024+256) | **50%** |

| 参数 | 值 | 说明 |

### 优势

1. **数学严格**：SVD 是最优低秩近似（Eckart-Young 定理），保留前 R 个奇异值最大化保留原始权重信息
2. **零运行时开销**：SVD 分解在模型加载时一次性完成，forward 时只是两次小 GEMM

## 方向一：MetaShuffling —— 基于 Token 排序的静态分派 + CUDA Graph 捕获

- **核心原理**：对每层 MoE 路由结果，按 expert ID 对 token 做原地排序（argsort + gather），使同一专家的 token 在内存中连续，从而把"变长分派"转化为"静态稠密 GroupedGEMM"。所有元数据（offset、专家计数）保留在 device 上，全程无 CPU-GPU 同步。
- **与传统方法不同**：传统 Dense SMoE batched GEMM 依赖 padding 到最大长度（浪费算力/显存）或 slicing（动态 shape 无法被 CUDA Graph 捕获）。MetaShuffling 通过"排序"使 activation 始终是定长稠密张量，因此**可以完整被 CUDA Graph 捕获**，消除上千次 kernel launch 开销。
- **预期加速**：单 GPU 端到端 1.5–2×，叠加 CUDA Graph 后 launch 开销从 ~50ms 降至 ~0.05ms。
- **实现复杂度**：中等。需重写 MoE forward：router→argsort→GroupedGEMM（cutlass/grouped）→gather_back。可借助 `torch.argsort` + `torch.ops.aten._grouped_mm`。
- **是否改权重**：否。
- **来源**：PyTorch 官方博客 "MetaShuffling: Accelerating Llama 4 MoE Inference"（2025-05）。

---

## 方向二：HookMoE —— 分层自适应 Top-k 削减 + 轻量补偿模块

- **核心原理**：观察发现 MoE 的**上层所需激活专家数显著少于下层**。将顶层几层的 top-2 降为 top-1，并插入一个极小的可训练 Hook 模块（单层线性）补偿精度损失。仅需 512 条校准样本、4 小时 8 卡训练 0.14–0.7% 参数。
- **与传统方法不同**：不是剪枝（不删除专家权重），不是蒸馏（不需要 teacher 完整 forward），而是"动态减少每 token 激活专家数 + 极小补偿头"。对 CTR 场景尤其合适——上层语义已稳定，路由冗余高。
- **预期加速**：激活专家数减半，prefill/forward 阶段 1.42× 加速；精度损失仅 2.5%。
- **实现复杂度**：中等。需在 infer.py 中加载额外 Hook 参数（可单独存为 hook.pt，不修改原 ckpt.pt），并修改顶层路由逻辑。
- **是否改权重**：**不修改原 ckpt.pt**，新增独立 Hook 参数文件（可视为推理侧插件）。
- **来源**：EMNLP 2025 "HookMoE"（南开大学），代码已开源 github.com/KerwinKai/HookMoE。

## 方向三：DeepFusionKernel —— SwiGLU/FFN 深度算子融合

- **核心原理**：将 MoE 专家 FFN 中的 `gate_matmul → up_matmul → SiLU → elementwise_mul → down_matmul` 四个独立 kernel 深度融合为**单个 kernel**，gate 与 up 投影共享同一输入 tile（L2 cache 复用），SiLU 在寄存器内完成，消除中间 HBM 读写。
- **与传统方法不同**：不是简单的"算子融合"（PyTorch 已融合部分 pointwise），而是**跨 GEMM 的深度融合**——让两个 GEMM 共享 A-tile 加载，把激活函数塞进 GEMM 的 epilogue。对带宽受限的小 batch 推理尤其有效。
- **预期加速**：H100 上 13.2%，A100 上 9.7%（vs SGLang SOTA）；对小 batch 推理更显著。
- **实现复杂度**：复杂。需写 Triton/CUTLASS kernel，或调用 `torch._inductor` 的融合策略。
- **是否改权重**：否。
- **来源**：arXiv:2602.11808 "Deep Kernel Fusion for Transformers"（Cambridge/Imperial，2026-02）。

## 方向四：Triton 持久化 + Cache-Aware Grouped GEMM

- **核心原理**：用 Triton 实现 Grouped GEMM，采用**持久化内核**（grid = SM 数，单 wave 完成全部 tile）+ **分组启动顺序**（grouped launch ordering）优化 L2 cache 命中。所有专家的 GEMM 在单 kernel 单 launch 内完成。
- **与传统方法不同**：不是"for 循环调 8 次 mm"（高 launch 开销），也不是简单 batched GEMM（不感知 cache）。持久化设计消除 wave quantization，cache-aware 排序提升 L2 命中率。
- **预期加速**：vs PyTorch 手动循环 1.42–2.62×（H100，DeepSeekv3 实测）。
- **实现复杂度**：中等。Triton 官方有 grouped_gemm tutorial 可直接改写，PyTorch 博客已开源实现。
- **是否改权重**：否。
- **来源**：PyTorch 博客 "Accelerating MoE's with a Triton Persistent Cache-Aware Grouped GEMM Kernel"（2025-08）。

## 方向五：Expert 缓存 + 路由预测（Hot Expert Caching）

- **核心原理**：CTR 场景下用户/广告特征分布相对稳定，专家激活呈现明显的"热门专家"长尾分布。离线统计每层 top-k 路由频率，**将高频专家权重常驻 GPU 显存的最快层（或预热到 L2）**，对低频专家按需加载；并训练一个极小的路由预测器，提前 prefetch 下一层将用到的专家。
- **与传统方法不同**：不是量化/压缩权重，而是**用访问模式优化绕开显存带宽瓶颈**。对 CTR 这种"路由可预测性强"的场景比通用 LLM 更有效。
- **预期加速**：SpecMoEOff 论文显示 expert weight 传输减少 80%+，端到端 2.5× decode 吞吐；单卡推理主要受益于 prefetch 隐藏的权重加载延迟。
- **实现复杂度**：中等。需离线 profile 路由分布 + 实现异步 prefetch（CUDA stream）。
- **是否改权重**：否。
- **来源**：arXiv:2508.21706 "SpecMoEOff"（南大，2025-08）；NVIDIA Dynamo 的 expert disaggregation 思路。

## 方向六：Mixture-of-Depths / 早期退出（Early Exit）

- **核心原理**：在 Transformer 中后层插入轻量 exit classifier（一个小 MLP + softmax），对"高置信度"的样本直接提前输出 logits→sigmoid，跳过剩余层。CTR 任务中大量"明显点击/明显不点击"的样本不需要跑满 8 层。
- **与传统方法不同**：不是静态剪层（会损失难样本精度），而是**按样本难度动态决定计算深度**。简单样本 3-4 层即可退出，难样本跑满 8 层。
- **预期加速**：平均层深度下降 30-50%，端到端 1.3-1.8×；AUC 损失可控（<0.5%）。
- **实现复杂度**：中等。需训练 exit classifier（小数据校准），在 infer.py 中实现逐层 confidence 判断 + 提前 return。
- **是否改权重**：**不修改原 ckpt.pt**，新增 exit head 参数（独立文件）。
- **来源**：Mixture-of-Depths (Raposo et al. 2024)；BranchyNet 系列；动态推理综述 TPAMI 2022。

## 方向七：MIND 自省式自适应计算（参数复用迭代）

- **核心原理**：ICLR 2025 Oral。引入一个 introspection network，根据输入复杂度**动态决定网络迭代多少次**（复用同一组 Transformer 参数），而非固定跑 8 层。简单输入 2-3 次迭代即达不动点，复杂输入多迭代。
- **与传统方法不同**：把"深度"从"层数"变成"迭代次数"，**参数量不变但有效计算量随难度自适应**。对 CTR 这种"特征交互可迭代精化"的任务天然契合。
- **预期加速**：论文显示 3 层网络达 ResNet-50 精度；Transformer 上 F1 几乎无损。预期 1.5-2× 加速。
- **实现复杂度**：复杂。需重新组织 forward 为"单层迭代 + 收敛判断"，并训练 introspection gate。
- **是否改权重**：**不修改原 ckpt.pt**，新增 introspection 参数。
- **来源**：ICLR 2025 Oral "MIND over Body: Adaptive Thinking using Dynamic Computation"。

## 方向八：CUDA Graph 静态捕获 + 多流并行 + 算子融合

- **核心原理**：将整个 `model forward + sigmoid` 用 `torch.cuda.CUDAGraph` 捕获为静态图，replay 时单次 launch。配合方向一的排序使 MoE 部分可捕获；attention/LayerNorm 等用 Triton 融合 kernel 替换；用独立 CUDA stream 让 router 计算与上一层 FFN 尾部重叠。
- **与传统方法不同**：不是 torch.compile（已被排除），而是**手动 CUDA Graph + 多流调度**——更细粒度控制，能处理 compile 难以优化的动态路由（通过排序静态化）。典型 Transformer forward 有 1000+ kernel launch，Graph 化后 launch 开销从 ~54ms 降至 ~0.05ms。
- **预期加速**：10-20%（launch 主导场景更高）；叠加多流重叠再 +5-10%。
- **实现复杂度**：中等。需 warmup + 静态输入 buffer + `graph.replay()`。注意固定输入地址（`copy_` 而非重新分配）。
- **是否改权重**：否。
- **来源**：PyTorch CUDA Graph 文档；TensorRT-LLM DeepSeek-R1 优化博客；NVIDIA CUDA 12.8 条件节点增强。

## V175-AUTOTUNE 新提交（max-autotune 编译模式）

### 设计思路

在 V174 基线上叠加 `torch.compile(mode="max-autotune-no-cudagraphs")`：

| 维度 | 分析 |
|---|---|
| **优化原理** | 让 PyTorch 在 warmup 阶段自动搜索每个 GEMM 的最优 cuBLAS 算法，V100 上预期 10-30% kernel 加速 |
| **AUC/PCOC 风险** | 零（纯执行优化，不改变数值语义） |
| **为何不用 reduce-overhead** | V112 已证明 reduce-overhead（含 CUDA Graphs）在动态 shape 下灾难性失败（1332s） |
| **为何不用 CUDA Graph** | V149 手动 CUDA Graph for SMoE 与 torch.compile(dynamic=True) 冲突——graph.replay() 导致 dynamo graph-break，每层每 batch ~0.3ms × 8层 × 150batch ≈ 0.36s 开销，超过 CUDA Graph 收益（~0.14s），反生产 |
| **max-autotune-no-cudagraphs** | 保留 autotune 算法搜索，去掉 CUDA Graphs，规避动态 shape 问题 |

### 安全回退机制

V175 在 warmup 路径（不计时）添加 try/except 三级回退：

1. 首选 `max-autotune-no-cudagraphs`
2. warmup 失败 → 回退 `default` 模式
3. 再失败 → 回退 eager（无 compile）

回退仅触发于 warmup 阶段，不影响正式计时的 batch。
