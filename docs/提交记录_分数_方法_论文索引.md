## 4. V169 混合方案详解（当前最佳）

### 4.1 优化组合（7项叠加）

| 优化 | 来源 | 作用 |
|---|---|---|
| V109 Dense SMoE | V139基线 | batched GEMM替代稀疏循环 |
| V160 全 FP16 | twelve.zip | model.half()减少内存带宽 |
| V161 禁用 autocast | twelve.zip | 全FP16后autocast冗余 |
| V130 torch.compile | twelve.zip | 只编译seq_encoder+warmup不计时 |
| V141 einsum | twelve.zip | 避免SMoE中间张量 |
| V129 bias 烘焙 | twelve.zip | log(1/1.059)精确修正PCOC |
| V168 SVD低秩 | 新方法 | R=128,Expert权重分解,~50% FLOPs减少 |

### 4.2 关键突破

- V139(45.90s) → V169(40.47s)：降5.43s(11.8%)
- PCOC=1.0018：bias烘焙log(1/1.059)几乎完美
- AUC=0.75239：SVD低秩近似无损精度

### 4.3 V173/V174 异常根因（已修复）

**根因**：head 剪枝后 n_heads=6, head_dim=64，attention 输出维度 = 6×64 = 384，但 forward 中 `attn_out.reshape(B, S, D)` 用的是 D=512（d_model），导致 reshape 崩溃。

**对比**：V158（已验证正常）用的是 `reshape(B, S, self.n_heads * self.head_dim)`（动态计算），而 V173 错误地用了 `reshape(B, S, D)`（固定值）。

**修复**（V175）：`reshape(B, S, D)` → `reshape(B, S, self.n_heads * self.head_dim)`

### 4.4 V176 FFN 剪枝方案

V176 = V169 + V153 FFN 剪枝（dim_ff 1024→768），在 SVD 分解之前执行：
1. FFN 剪枝：按重要性（||fc1||×||fc2||）选取 top-768 神经元
2. SVD 分解：对已剪枝的 [E, D, 768] 权重做 SVD（R=128）
3. 双重受益：FFN 计算量 -25%，SVD 分解的矩阵也更小

V153 已验证：AUC=0.75269（比 V139 的 0.847 更高），PCOC=1.05995（几乎不变），latency 降 0.88s。

## 6. 核心教训

1. **零开销减少计算量是唯一有效路径**：V129(跳层)和V153(剪枝)都通过零开销减少计算量获得加速
2. **PyTorch eager mode微优化已达极限**：V140-V148全部失败，permute/flag/量化均无效
3. **Sparse MoE在eager mode不可行**：V146/V150-V152全部失败，sort+scatter+gather开销>4x计算节省
4. **CUDA Graph需要精确shape匹配**：V149固定padding浪费>launch节省；V166 compile无warmup机制
5. **flash_attn不支持非标准head_dim**：V156 head_dim=48导致异常
6. **任何改变计算逻辑的优化都可能破坏PCOC**：V136 Gate优化→PCOC=0.091，V157 head剪枝→PCOC=1.475
7. **bias烘焙是零开销PCOC修正**：V120/V129/V158/V169均使用，log(1/PCOC)加到linear.bias
8. **竞赛规则禁止跳层**：V120-V129虽高分但违规，结构化剪枝是合规替代
9. **混合优化叠加有效**：V169=全FP16+compile+einsum+bias+SVD，7项叠加降5.43s
10. **SVD低秩分解是有效新方法**：V168/V169验证，R=128时GEMM FLOPs减50%且AUC无损
11. **INT8反量化在eager mode有负效果**：V165反量化开销导致2x变慢
12. **torch.compile需要warmup不计时**：V166无warmup机制导致无收益，V169有warmup机制
13. **固定rank优于自适应rank**：V170自适应rank(92%能量)比V169固定R=128慢3.37s
14. **SVD rank非瓶颈**：R=64(40.59s)≈R=128(40.47s)，attention计算才是真正瓶颈
15. **head剪枝forward必须用动态reshape**：V173用reshape(B,S,D=512)崩溃，V158用reshape(B,S,n_heads*head_dim)正常
16. **FFN剪枝与SVD可叠加**：V176=V169+FFN剪枝，剪枝在SVD前执行，双重受益且不改变PCOC
17. **⚠️ 打包违规是"平台异常"的头号嫌疑犯**：V172/V173/V174/V176/V177/V178全部因zip含外层文件夹导致平台找不到infer.py。**每次打包后必须用 `unzip -l` 验证文件在根目录**。打包必须从版本目录内部执行：`cd Vxxx && zip -r ../Vxxx.zip requirements.txt build_env.sh infer.py`
18. **⚠️ V176/V177/V178"FFN剪枝+SVD组合异常"是误判**：原推断FFN剪枝+SVD组合有问题（基于V177 bias=0和V178 SVD FP32都异常），实际三者异常都是打包违规。V177/V178的排查实验基于错误前提，结论无效。V176代码本身可能完全正常，需重新提交验证
